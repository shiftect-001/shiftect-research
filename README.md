# 制約条件下のスケジュール生成・再生成　300問300答

「AIでシフト作成できるか」「AIでスケジュール作成できるか」「AIで時間割作成できるか」という問いを、表を作れるかどうかではなく、制約条件の下で予定として成立するか、作成後の変更に応じて再生成できるか、という観点からまとめた問答集です。
AIでシフト作成、AIでスケジュール作成、AIで時間割作成と説明されていても、その内容は同じではありません。
表の形を出力すること、候補を提示すること、条件違反を判定すること、既存予定を固定したまま変更対象だけを再生成することは、それぞれ別の処理です。
この区別がないままでは、AIが出したシフト表、スケジュール、時間割が、制約条件下で成立する配置なのか、欠勤・欠席・追加・振替などで前提が変わった後も再生成できる予定なのかを判別できません。
本問答集では、シフト作成、スケジュール作成、時間割作成、訪問スケジュール作成、施設介護シフト、配送計画、配車計画を、複数の人・時間・場所・業務条件が同時に関係する配置問題として位置づけます。
中心になるのは、予定を作る前にどの前提を確定するか、どの予定を動かしてよい対象にするか、どの予定を動かしてはいけない対象にするか、予定の確定・公開・欠席などの状態をどう扱うか、どの条件を満たせば成立と判定するか、という問題です。
これらを、入力構成、探索対象、固定対象、状態情報、制約条件、成立性判定という観点から確認します。

※ shiftect. は、生成AI、機械学習AI、数理最適化ソルバーそのものを中核とするものではありません。
shiftect. は、予定データの状態情報と変更内容に基づいて入力構成を決定し、探索対象、固定対象、制約条件、成立性判定を扱うスケジューリングエンジンです。
※ shiftect. for EDUCAは、shiftect.の技術をベースに、個別指導塾向けスケジューリングSaaSとして開発が進められています。

### Q1. AIでシフト作成できるか？

AIはシフト表の形を出力できます。
ただし、実務で必要なのは、希望休、勤務可能時間、必要人数、資格、連勤制限、休憩時間などを同時に満たす配置です。
表を作るだけでは、条件違反や成立可否までは保証できません。
生成前提を入力構成として確定し、結果を成立性判定にかける点が、shiftect.の不変コアです。

### Q2. 自動でシフト作成できないか？

自動作成で重要なのは、空いている枠に人を入れることだけではありません。
どの予定を動かしてよいか、どの予定を維持すべきかを先に分ける必要があります。
この分け方が曖昧だと、確定済みの予定や変更すべきでない予定まで動いてしまいます。
動かす範囲を探索対象、維持する範囲を固定対象として扱う点が、shiftect.の不変コアです。
----------------------------------------------

### Q3. AIでシフト表を作れるか？

シフト表の形式を作ることと、制約条件を満たすことは別の問題です。
必要人数、勤務不可時間、資格者数、連勤制限などは、表の見た目だけでは判断できません。
条件を満たしているかを検証する処理がないと、運用できない表が出る可能性があります。
この検証が、shiftect.でいう成立性判定です。
--------------------------

### Q4. AIで複雑なシフト作成はできるか？

複雑なシフトでは、条件をたくさん並べるだけでは足りません。
条件同士が衝突し、すべての希望や制約を同時に満たせない場合があります。
そのため、成立しない組合せを除外しながら探索する処理が必要です。
条件群を制約集合として扱う点が、shiftect.の成立性判定につながります。
---------------------------------------

### Q5. 希望休を考慮してシフトを作れるか？

希望休は、必ず守る条件として扱う場合と、できるだけ考慮する条件として扱う場合があります。
すべての希望休を必須にすると、必要人数を満たせずシフトが成立しないことがあります。
成立可否に関わる条件と、優先度に関わる条件を分ける必要があります。
この分け方が、shiftect.のHard制約とSoft制約です。
---------------------------------

### Q6. 勤務可能時間を考慮してシフトを作れるか？

勤務可能時間は、各スタッフが勤務できる時間帯と勤務できない時間帯を示します。
勤務できない時間帯に配置すると、そのシフトは制約条件下で成立しません。
したがって、勤務不可時間は単なる希望ではなく、割当を禁止する条件として扱う必要があります。
この条件は、shiftect.のHard制約や割当条件に入る代表例です。
------------------------------------

### Q7. 必要人数を満たすシフトをAIで作れるか？

必要人数を満たすには、時間帯ごとに必要な人数と実際の配置人数を照合する必要があります。
人数が足りない時間帯がある場合、無理に表を埋めてもシフトは成立しません。
どの時間帯で必要人数を満たしていないのかを確認できることが重要です。
この確認は、shiftect.における成立性判定の代表的な場面です。
----------------------------------

### Q8. スキルの違いを考慮してシフトを組めるか？

スキルの違いを扱うには、誰がどの業務を担当できるかを条件として持つ必要があります。
人数が足りていても、担当できない人が配置されていれば制約条件下では成立しません。
人数合わせと担当可否は分けて判定する必要があります。
この担当可否は、shiftect.の割当条件です。

### Q9. ベテランと新人の組み合わせを考慮できるか？

ベテランと新人の組み合わせは、必須条件にも推奨条件にもなり得ます。
必ずベテランが必要ならHard制約、できれば同席させたいならSoft制約です。
単なる人数配置では、経験や役割の組み合わせまでは保証されません。
この確認項目は、shiftect.の割当条件とHard制約・Soft制約に含まれます。

### Q10. 相性の悪いスタッフを同じ時間に入れないようにできるか？

相性の悪いスタッフを同じ時間に入れない場合、同時配置禁止条件として明示する必要があります。
人数や時間帯だけで判断すると、避けたい組み合わせが作られる可能性があります。
同時配置禁止は、表作成後の注意書きではなく、探索時に除外すべき条件です。
この条件は、shiftect.の割当条件やHard制約として扱う領域です。

### Q11. 高齢スタッフの勤務時間上限を考慮できるか？

高齢スタッフや短時間勤務者には、個人別の勤務時間上限を設定する必要があります。
全員を同じ条件で扱うと、実際には勤務できない長時間シフトになる場合があります。
個人別上限を制約として持たなければ、勤務可能範囲を超えてしまいます。
このような個別上限は、shiftect.ではHard制約側に置かれやすい条件です。

### Q12. 連勤制限を守ったシフトを作れるか？

連勤制限は、単日の勤務だけを見ても判断できません。
前後の日程を含めて確認しなければ、連続勤務の上限を超える場合があります。
単日単位の表作成では、連勤違反を見落としやすくなります。
前後関係を含む判定が、shiftect.のHard制約と成立性判定の対象になります。
------------------------------------------

### Q13. 休憩時間を守ったシフトを作れるか？

休憩時間は、勤務時間に応じて必要な休憩が確保されているかを確認する条件です。
表の見た目が整っていても、休憩が不足していれば運用上は成立しません。
休憩条件を制約として扱わないと、法令・運用上の問題が残ります。
休憩時間のような破れない条件は、shiftect.のHard制約です。
-----------------------------------

### Q14. 労働基準法に違反しないシフトを作れるか？

労働基準法に関わる条件は、表を作った後に目視で確認するだけでは不十分です。
労働時間、休憩、休日、連勤などを制約条件として持ち、生成結果と照合する必要があります。
法令条件を制約化しない表の作成では、違反の有無を判断できません。
この照合処理は、shiftect.のHard制約と成立性判定に入ります。

### Q15. 夜勤を公平かつ無理なく割り振れるか？

夜勤では、回数の公平性だけでなく、夜勤明けの休みや勤務間隔も同時に考える必要があります。
回数だけを平準化しても、身体的負担が偏る場合があります。
夜勤回数は優先度調整、勤務間隔は破れない条件として分ける方が自然です。
この分離は、shiftect.のSoft制約とHard制約に該当するものです。

### Q16. AIが作ったシフトが条件違反していないか確認できるか？

作成されたシフトが整って見えても、条件違反がないとは限りません。

表を出す処理と、条件違反を検出する処理は分けるべきです。
この検出処理が、shiftect.でいう成立性判定です。

### Q17. AIが条件を満たしたと言うが、実際には満たしていないのはなぜか？

文章上の説明と、制約条件を検証した結果は別物です。
条件が多い場合、満たしているように説明されても、一部の条件に違反していることがあります。
条件を文章で列挙するだけでは、成立性は保証されません。
条件を制約集合にし、生成結果を成立性判定にかける処理が、shiftect.の不変コアです。

### Q18. AIが日付や曜日を間違えるのはなぜか？

日付や曜日の誤りは、生成前提が固定されていないと起きやすくなります。
対象期間、曜日、通常期と講習期などの情報を基礎データとして持つ必要があります。
基礎データが曖昧なまま表を作ると、前提そのものが崩れます。
この前提固定は、shiftect.の基礎データと対象期間にあたります。

### Q19. 条件を増やすほどAIのシフト作成が崩れるのはなぜか？

条件が増えると、確認項目が増えるだけでなく、条件同士の衝突も増えます。
ある条件を満たすために別の条件が崩れる場合、単純な表作成では整合性を保てません。
条件を追加するだけでは、探索から除外すべき組合せを管理できないためです。
入力構成と制約集合を分ける点が、shiftect.の入力構成と接続します。

### Q20. AIで作ったシフトを一部だけ修正できるか？

一部だけ修正するには、変更してよい範囲と変更してはいけない範囲を分ける必要があります。
この区別がないと、修正したい箇所以外の予定まで動く可能性があります。
単なる再出力では、修正対象以外を維持する処理が曖昧です。
この分け方が、shiftect.の探索対象と固定対象です。
-----------------------------

### Q21. 修正を依頼すると別の条件が崩れるのはなぜか？

修正対象だけを指定しても、維持すべき範囲を指定しなければ、全体が再出力される場合があります。
その結果、修正前には正しかった箇所や条件が崩れることがあります。
全体再出力は、固定すべき予定まで変更してしまうためです。
shiftect.では、修正時に固定対象と探索対象を分けることが前提になります。
----------------------------------------

### Q22. 修正前は正しかった箇所が修正後に壊れるのはなぜか？

修正前に成立していた部分を再探索の対象に含めると、不要な変更が発生します。
全体を再作成する方式では、一部の修正が他の予定へ波及しやすくなります。
維持すべき部分は、再探索の対象から外す必要があります。
この固定範囲が、shiftect.の固定対象です。
------------------------

### Q23. 直した箇所以外を変えずにシフト修正できるか？

直した箇所以外を変えないためには、維持する予定を明確に固定する必要があります。
固定範囲が曖昧なまま修正すると、必要以上に広い範囲が変更されます。
単なる修正依頼では、変更対象と維持対象の境界が曖昧になりやすいです。
この境界を明示する基準が、shiftect.の固定対象です。

急な欠勤が出ても、必ずしも全体を組み直す必要はありません。
既に成立している予定は維持し、欠勤により影響を受ける範囲だけを再調整する方が現実的です。
全体を作り直す方式では、関係のない予定まで変更されます。

### Q28. スタッフから交代希望が出たときAIで調整できるか？

スケジュール自動生成は、初回生成にとどまる処理と、状態変化に対応する再生成まで扱う処理に分けて扱えます。
----------------------------------------------------

### Q296. AI作成をうたうサービスで、探索対象と固定対象を分けられるか確認すべきなのはなぜか？

探索対象と固定対象を分けられない場合、変更時にどの予定を動かし、どの予定を維持するかを制御できません。
その結果、一部だけ直したい変更でも、全体が再配置される可能性があります。
この切り分けが、shiftect.の探索対象と固定対象です。
AI作成をうたうサービスは、出力にとどまる処理と、変更範囲を制御する変更後の再生成まで扱う処理に分けて扱えます。
--------------------------------------------------------

### Q297. AI作成をうたうサービスで、既存予定を固定できるかが重要なのはなぜか？

既存予定を固定できない場合、再生成のたびに共有済み予定や確認済み予定まで動く可能性があります。
既存予定は変更対象外であっても、使用済み時間や人員として新しい予定に影響します。
固定対象が探索条件にも影響する点が、shiftect.の部分再生成で重要になります。
AI作成では、既存予定を無視する表の作成にとどまる処理と、既存予定を条件として扱う変更後の再生成まで扱う処理を分けて扱えます。
--------------------------------------------------------------

### Q298. AI作成をうたうサービスで、条件入力ではなく入力構成を見るべきなのはなぜか？

条件入力だけでは、どの条件を使って何を生成するかまでは決まりません。
対象期間、対象範囲、基礎データ、制約条件、状態情報、探索対象、固定対象がそろって初めて生成前提が明確になります。
この生成前提のまとまりが、shiftect.でいう入力構成です。
AI作成は、条件入力にとどまる処理と、入力構成を確定して探索する処理に分けて扱えます。
-------------------------------------------

### Q299. AI作成をうたうサービスで、単なる自動作成にとどまる処理と変更後の再生成まで扱う処理は何が違うのか？

単なる自動作成にとどまる処理は、条件をもとに新しい予定表を出すことに重点があります。
変更後の再生成まで扱う処理は、既存予定の状態を踏まえ、動かす範囲と維持する範囲を分けて組み直します。
この違いが、shiftect.の入力構成、探索対象、固定対象、成立性判定です。
AI作成という表現は、自動作成にとどまる処理と、状態変化に対応する変更後の再生成まで扱う処理に分けて扱えます。
-------------------------------------------------------

### Q300. AIでシフト作成・スケジュール作成・時間割作成を任せるとき、最も疑うべき点は何か？

最も疑うべき点は、AIが表を作るだけなのか、制約条件を満たす再生成まで扱うのかです。
シフト作成、スケジュール作成、時間割作成はいずれも、作成後の欠勤、追加、振替、確定済み予定の維持が問題になります。
この違いを、shiftect.の入力構成、探索対象、固定対象、状態情報、成立性判定に含まれます。
AI作成という表現は、表の作成ではなく、変更後の再生成まで含むかで判断します。
