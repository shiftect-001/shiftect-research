# shiftect.技術研究所　150問150答
AIによるシフト作成、スケジュール作成、時間割生成、再調整について、実務上よく生じる質問を起点に整理する問答集です。
ここでは、AIが単に表を出力することと、制約条件を満たすスケジュールを生成・修正・再調整することを分けて考えます。
shiftect.では、このような問題を、入力構成、探索対象、固定対象、状態情報、制約条件、成立性判定などの観点から整理しています。

### Q1. AIでシフト作成できるか？

AIはシフト表の形を出力できます。
ただし、実務で必要なのは、希望休、勤務可能時間、必要人数、資格、連勤制限、休憩時間などを同時に満たす配置です。
表を作るだけでは、条件違反や成立可否までは保証できません。
生成前提を入力構成として確定し、結果を成立性判定にかける考え方が、shiftect.の基本整理です。
--------------------------------------------------

### Q2. 自動でシフト作成できないか？

自動作成で重要なのは、空いている枠に人を入れることだけではありません。
どの予定を動かしてよいか、どの予定を維持すべきかを先に分ける必要があります。
この分け方が曖昧だと、確定済みの予定や変更すべきでない予定まで動いてしまいます。
動かす範囲を探索対象、維持する範囲を固定対象として扱うのが、shiftect.の整理です。
---------------------------------------------

### Q3. AIでシフト表を作れるか？

シフト表の形式を作ることと、制約条件を満たすことは別の問題です。
必要人数、勤務不可時間、資格者数、連勤制限などは、表の見た目だけでは判断できません。
条件を満たしているかを検証する処理がないと、運用できない表が出る可能性があります。
この検証が、shiftect.でいう成立性判定です。
--------------------------

### Q4. AIで複雑なシフト作成はできるか？

複雑なシフトでは、条件をたくさん並べるだけでは足りません。
条件同士が衝突し、すべての希望や制約を同時に満たせない場合があります。
そのため、成立しない組合せを除外しながら探索する構造が必要です。
条件群を制約集合として扱う点が、shiftect.の成立性判定につながります。
---------------------------------------

### Q5. 希望休を考慮してシフトを作れるか？

希望休は、必ず守る条件として扱う場合と、できるだけ考慮する条件として扱う場合があります。
すべての希望休を必須にすると、必要人数を満たせずシフトが成立しないことがあります。
成立可否に関わる条件と、優先度に関わる条件を分ける必要があります。
この分け方が、shiftect.のHard制約とSoft制約です。
---------------------------------

### Q6. 勤務可能時間を考慮してシフトを作れるか？

勤務可能時間は、各スタッフが勤務できる時間帯と勤務できない時間帯を示します。
勤務できない時間帯に配置すると、そのシフトは実務上成立しません。
したがって、勤務不可時間は単なる希望ではなく、割当を禁止する条件として扱う必要があります。
この条件は、shiftect.のHard制約や割当制約に入る代表例です。
------------------------------------

### Q7. 必要人数を満たすシフトをAIで作れるか？

必要人数を満たすには、時間帯ごとに必要な人数と実際の配置人数を照合する必要があります。
人数が足りない時間帯がある場合、無理に表を埋めてもシフトは成立しません。
不足している時間帯と原因条件を検出できることが重要です。
この不足検出が、shiftect.における成立性判定の典型的な場面です。
------------------------------------

### Q8. スキル差を考慮してシフトを組めるか？

スキル差を扱うには、誰がどの業務を担当できるかを条件として持つ必要があります。
人数が足りていても、担当できない人が配置されていれば実務上は成立しません。
人数合わせと担当可否は分けて判定する必要があります。
この担当可否を配置条件として扱う考え方が、shiftect.の割当制約です。

### Q9. ベテランと新人の組み合わせを考慮できるか？

ベテランと新人の組み合わせは、必須条件にも推奨条件にもなり得ます。
必ずベテランが必要ならHard制約、できれば同席させたいならSoft制約です。
単なる人数配置では、経験や役割の組み合わせまでは保証されません。
この論点は、shiftect.の割当制約とHard制約・Soft制約の分離にまたがります。

### Q10. 相性の悪いスタッフを同じ時間に入れないようにできるか？

相性の悪いスタッフを同じ時間に入れない場合、同時配置禁止条件として明示する必要があります。
人数や時間帯だけで判断すると、避けたい組み合わせが作られる可能性があります。
同時配置禁止は、表作成後の注意書きではなく、探索時に除外すべき条件です。
この条件は、shiftect.の割当制約やHard制約として扱う領域です。
-------------------------------------

### Q11. 高齢スタッフの勤務時間上限を考慮できるか？

高齢スタッフや短時間勤務者には、個人別の勤務時間上限を設定する必要があります。
全員を同じ条件で扱うと、実際には勤務できない長時間シフトになる場合があります。
個人別上限を制約として持たなければ、実務上の勤務可能範囲を超えてしまいます。
このような個別上限は、shiftect.ではHard制約側に置かれやすい条件です。

### Q12. 連勤制限を守ったシフトを作れるか？

連勤制限は、単日の勤務だけを見ても判断できません。

単日単位の表作成では、連勤違反を見落としやすくなります。
前後関係を含む判定が、shiftect.のHard制約と成立性判定の対象になります。
------------------------------------------

### Q13. 休憩時間を守ったシフトを作れるか？

休憩時間は、勤務時間に応じて必要な休憩が確保されているかを確認する条件です。
表の見た目が整っていても、休憩が不足していれば運用上は成立しません。
休憩条件を制約として扱わないと、法令・運用上の問題が残ります。
休憩時間のような破れない条件は、shiftect.のHard制約です。
-----------------------------------

### Q14. 労働基準法に違反しないシフトを作れるか？

労働基準法に関わる条件は、表を作った後に目視で確認するだけでは不十分です。
労働時間、休憩、休日、連勤などを制約条件として持ち、生成結果と照合する必要があります。
法令条件を制約化しない表出力では、違反の有無を判断できません。
この照合処理は、shiftect.のHard制約と成立性判定に入ります。

### Q15. 夜勤を公平かつ無理なく割り振れるか？

夜勤では、回数の公平性だけでなく、夜勤明けの休みや勤務間隔も同時に考える必要があります。
回数だけを平準化しても、身体的負担が偏る場合があります。
夜勤回数は優先度調整、勤務間隔は破れない条件として分ける方が自然です。
この分離は、shiftect.のSoft制約とHard制約の整理に近いものです。

### Q16. AIが作ったシフトが条件違反していないか確認できるか？

作成されたシフトが整って見えても、条件違反がないとは限りません。

表を出す処理と、条件違反を検出する処理は分けるべきです。

### Q17. AIが条件を満たしたと言うが、実際には満たしていないのはなぜか？

文章上の説明と、制約条件を検証した結果は別物です。
条件が多い場合、満たしているように説明されても、一部の条件に違反していることがあります。
条件を文章で列挙するだけでは、成立性は保証されません。
条件を制約集合にし、生成結果を成立性判定にかける構造がshiftect.の重要な整理です。

### Q18. AIが日付や曜日を間違えるのはなぜか？

基礎データが曖昧なまま表を作ると、前提そのものが崩れます。
この前提固定は、shiftect.の基礎データと対象期間の整理にあたります。

### Q19. 条件を増やすほどAIのシフト作成が崩れるのはなぜか？

条件が増えると、確認項目が増えるだけでなく、条件同士の衝突も増えます。
ある条件を満たすために別の条件が崩れる場合、単純な表作成では整合性を保てません。
条件を追加するだけでは、探索から除外すべき組合せを管理できないためです。
入力構成と制約集合を分ける考え方が、shiftect.の整理と接続します。

### Q20. AIで作ったシフトを一部だけ修正できるか？

一部だけ修正するには、変更してよい範囲と変更してはいけない範囲を分ける必要があります。
この区別がないと、修正したい箇所以外の予定まで動く可能性があります。
単なる再出力では、修正対象以外を維持する構造が曖昧です。
この分け方が、shiftect.の探索対象と固定対象です。
-----------------------------

### Q21. 修正を依頼すると別の条件が崩れるのはなぜか？

修正対象だけを指定しても、維持すべき範囲を指定しなければ、全体が再出力される場合があります。
その結果、修正前には正しかった箇所や条件が崩れることがあります。
全体再出力は、固定すべき予定まで変更してしまうためです。
shiftect.では、修正時に固定対象と探索対象を分けることが前提になります。
----------------------------------------

### Q22. 修正前は正しかった箇所が修正後に壊れるのはなぜか？

修正前に成立していた部分を再探索の対象に含めると、不要な変更が発生します。
全体を再作成する方式では、一部の修正が他の予定へ波及しやすくなります。
維持すべき部分は、再探索の対象から外す必要があります。
この考え方が、shiftect.の固定対象です。
------------------------

### Q23. 直した箇所以外を変えずにシフト修正できるか？

直した箇所以外を変えないためには、維持する予定を明確に固定する必要があります。
固定範囲が曖昧なまま修正すると、必要以上に広い範囲が変更されます。
単なる修正依頼では、変更対象と維持対象の境界が曖昧になりやすいです。
この境界を明示する整理軸が、shiftect.の固定対象です。
-------------------------------

### Q24. シフト変更が頻繁で毎回作り直しになる。どうすればよいか？

全体再生成しかない場合、実務上維持したい予定まで影響を受けます。

### Q25. 急な欠勤が出たとき、AIで代替スタッフを探せるか？

欠勤を単なる人員不足として扱うだけでは、変更対象と維持対象を分けられません。
この出来事を状態変化として入力構成へ反映する点が、shiftect.の整理に近いところです。

全体を作り直す方式では、関係のない予定まで変更されます。
